ECHO

Technical Architecture &
AI Systems Design

Echo is a spoiler-safe companion platform for television series: it continuously harvests real viewer conversation from public platforms, aligns every comment to the exact scene it talks about, and replays that global conversation to each viewer strictly according to where they are in the story — while an AI moderator keeps the room alive. This document describes the production system as deployed today: the full service topology, the data pipeline, the four LLM-powered engines at the core of the product, and how each user-facing capability is realized.

Live — public beta on Azure Stack: .NET 10 · Next.js 16 / React 19 · PostgreSQL 18 + pgvector · SignalR AI: gpt-4.1-mini + text-embedding-3-small Document date: July 2026

01 · Executive overview

What Echo does, in one pass

Watching a series a few days late usually means muting every social platform. Echo inverts that: it brings the conversation to the viewer, filtered by narrative time. Every channel is bound to a series season or a single episode. The viewer sets (or plays) a watch position — "episode 1, minute 43" — and the feed shows only what is safe at that exact moment: curated comments harvested from YouTube, X, Bluesky and Trakt that were aligned to earlier scenes, plus the room's live chat, with anything ahead of the viewer's position protected behind a blur they can consciously reveal.

Four AI engines make this possible: an Enrichment engine that deduplicates, scene-aligns, spoiler-scores and translates every harvested comment; a Segment Intelligence engine that converts each episode into a timed, gapless scene map (the coordinate system everything else relies on); a Realtime Spoiler Shield that classifies live user messages the moment they are written; and a Moderator AI that hosts each room with presence-aware pacing and private, position-safe replies. All four run on two OpenAI foundation models (gpt-4.1-mini for JSON-mode reasoning, text-embedding-3-small for 1536-dimension vectors) behind a single resilient client with retry/backoff.

02 · System topology

Service map — one VM, clean seams

Echo runs as four containers behind a Caddy reverse proxy on a single Azure VM (public beta scale), with clean seams that allow each tier to scale out independently later. The Next.js frontend is a PWA — installable to the home screen on iOS/Android with a pinned, zoom-locked viewport so it behaves like a native app. The .NET 10 API serves REST + SignalR WebSockets; background workers run the harvesting and AI pipelines; PostgreSQL 18 with the pgvector extension is the single source of truth — relational rows, vectors and full state machines in one database.

Fig. 1 — Production topology. Animated particles trace the hot paths: user traffic (red), relational + vector persistence (blue), harvested comments (green), LLM calls (amber).
Frontend

Next.js 16 (Turbopack) + React 19, Tailwind v4 design system in pure black / Echo-red, framer-motion micro-interactions, Radix primitives. Ships as a PWA: programmatic icon set, standalone display, portrait lock, pinch/double-tap zoom disabled at the viewport level while every scroll surface stays free.

Backend

ASP.NET Core (.NET 10) minimal-API endpoints with strict request validation, JWT (Google Sign-In exchange), EF Core 10 with snake_case mapping, UUIDv7 keys, UTC timestamps and xmin optimistic concurrency. SignalR hub carries all room-level realtime traffic.

Workers

Independent hosted services per concern: source collectors (content-scoped, quota-aware with per-source cooldowns), the enrichment batch loop, channel preparation, and the moderator loop. A worker crash never touches the API's availability.

03 · Ingestion & curation

From raw internet noise to a scene-aligned pool

Harvesting is content-scoped, not channel-scoped: collectors follow a series (its search queries, videos, posts and Trakt episode threads) regardless of which channels are currently open, so the pool keeps growing even while rooms are closed. Each raw comment enters an explicit nine-state machine — every failure is visible, idempotent and resumable. The unique key (content, platform, external-id) makes at-least-once collection safe: re-fetching the same comment can never duplicate it.

Fig. 2 — The curation conveyor. Roughly one in five raw comments survives to publication; the rest are duplicates folded into a "×N say the same" counter, filtered noise, or visible failures.

Duplicate handling has product value beyond hygiene: when a new comment lands inside an existing semantic cluster (cosine distance under threshold against the cluster centroid), it is not published twice — instead the original curated message's SimilarCount increments, so the feed can honestly show "hundreds of viewers said exactly this" without repeating itself.

04 · AI layer

Four engines, two foundation models

Every AI capability in Echo is a thin, auditable JSON contract on top of the same two models. There is deliberately no fine-tuning and no agent framework in the loop: each engine builds a rich, factual context from the database (segments, positions, presence), asks one strictly-shaped JSON question, validates the response and stores the verdict. Retries with exponential backoff live in a single shared OpenAI client; each engine defines its own failure semantics (fail-open, fail-visible, or skip).

05 · Engine 1

Enrichment — one call that does three jobs

For every deduplicated survivor the engine makes exactly one gpt-4.1-mini call that answers three questions at once — keeping cost linear and the contract auditable. The prompt carries the series title, the aired episode list and the three nearest scene segments found by pgvector cosine search over the comment's own embedding; the model must either snap the comment to one of those candidates or declare it "general".

🎯Scene alignment

Returns episode + anchorSeconds (segment midpoint) when the comment clearly refers to a candidate scene; otherwise both are null and the comment becomes position-independent "general" talk. This anchor is what the timeline slider filters on.

☠️Absolute spoilers

isAbsoluteSpoiler marks twist/death/ending reveals that are unsafe at any position (e.g. a finale outcome quoted in an episode-1 discussion), plus a 0–1 confidence score. These stay protected regardless of the slider.

🌐Faithful translation

Every message is stored bilingually (TextEn + TextTr, slang and tone preserved). The feed then renders in the user's profile language — one pool serves both audiences with zero extra calls at read time.

Concurrency discipline. DbContext is not thread-safe, so all database phases run sequentially; only the pure-HTTP LLM calls fan out under a semaphore (bounded parallelism). This keeps the loop fast without a single shared-state race.
06 · Engine 2

Segment Intelligence — the coordinate system

Everything position-aware in Echo (alignment, the shield, the wave, moderator scoping) depends on one artifact: a gapless timed scene map per episode, 8–12 segments covering second 0 to the exact runtime. The engine merges two sources with a strict precedence rule: the episode's SRT/VTT transcript timestamps are ground truth for WHEN; Wikipedia plot text (episode page → season page → series page fallback) and the curated context pack tell it WHAT happens. Summaries are deliberately explicit — names, deaths, twists — because they are the matching surface the other engines embed against.

Fig. 4 — Scene-map generation. A post-processing normalizer forces segments to be consecutive and gapless before persisting, so downstream code can trust the map blindly.

Freshly-aired episodes are a special case: Wikipedia may not have a plot section yet. The engine then degrades gracefully — curated context text (produced by the Operations Copilot from Trakt + Wikipedia research and approved by the admin) or the transcript alone drives the map, and the source used is recorded with the result.

07 · Engine 3

Realtime Spoiler Shield — classify, never censor

The instant a user submits a chat message (or answers a room question), the shield builds the channel's scope rules, attaches the season's full timed scene map, and asks for a strict verdict: {isSpoiler, reason}. The message is never blocked — it is flagged, and the UI blurs it for exactly the viewers whose own position makes it dangerous. A viewer taps to reveal; the choice is theirs. If the model call fails, the shield fails open: chat latency and availability always win.

Fig. 5 — Scope rules given verbatim to the model. Green = watched territory, red (blinking) = spoiler territory relative to the writer.

On the reader's side, protection is applied per viewer at render time: flagged messages arrive in everyone's payload with their flags (they are never dropped server-side for public messages), and the client blurs a message when the reader's spoiler-free mode is on — including any live message written from a position ahead of the reader's own. One classification, many personalized presentations.

08 · Engine 4

Moderator AI — a host, not a bot

Every live channel has a resident moderator persona ("Echo Moderator") run by a worker loop. Its defining property is restraint: the decision to speak passes through a strict hierarchy where cheap deterministic checks veto expensive reasoning, and the LLM is only consulted when speaking is even permissible. Message dosage adapts to presence — a packed room gets a quieter host; a lonely early viewer gets more company. Anti-flood cooldowns and per-hour caps are hard gates, not suggestions.

Fig. 6 — The decision funnel. Deterministic layers veto early and often; the model is the last, not the first, resort.

The @echo mention flow is the moderator's sharpest trick: when a user asks it something directly, the reply is delivered privately — persisted with a visible-to-user marker and pushed only to that user's personal SignalR group, so the payload never even reaches other clients. Crucially, the reply's knowledge is clamped to the asker's watch position: the prompt receives only scene summaries up to that point, making it structurally unable to spoil what the asker hasn't seen.

09 · Position-aware timeline

The slider, the wave, and time-travel chat

The channel player bar is the product's signature control. The viewer drags (or plays, 1 s/s with pause) a position across the episode/season; the curated feed expands in real time as scenes fall behind the playhead. The slider's visual layer is the Comment Wave — a bar chart of real anchor density per time bucket computed from the pool (no synthetic data), so the crowd's loudest moments are visible before you reach them, without revealing why.

🎚️Position mechanics

Client keeps an authoritative position ref; drags debounce (400 ms) before committing to the server, a ticker advances it while playing, and a 10 s poll + reconnect refresh reconciles state. Messages ahead of the position simply don't render from the curated pool; live chat ahead of it arrives blurred.

📊Wave data honesty

Buckets are aggregated from real curated anchors per episode (season channels concatenate aired episodes on a cumulative axis). Empty pool → flat line, never a fake pulse. The wave endpoint is scope-aware, so an episode channel only ever exposes its own episode's density.

10 · Realtime layer

SignalR fan-out — one hub, two audiences

All room realtime rides a single authenticated SignalR hub with two group classes per channel: the room group (every connected viewer) and a personal group per user, which exists so that private payloads — the moderator's @echo replies today, personal replies tomorrow — are never broadcast and filtered client-side, but simply never leave the server for anyone else. Presence (the live viewer count) is a by-product of hub connect/disconnect tracking. Clients reconnect with backoff and reconcile missed events through the regular REST reads, so a dropped socket degrades to 10-second polling instead of a broken room.

11 · Feature walkthroughs

How each capability is realized

🗳️Polls & room questions

Two first-class message kinds beside plain text. Polls: user-authored options (2–6), one vote per user enforced by a unique index, vote switching is an upsert, live percentages via PollChanged. Questions: upvote toggles, threaded answers that pass through the Spoiler Shield with the answerer's position, blurred for readers where needed.

🧨Reactions & emoji bursts

Two distinct affordances: reactions are persistent per-message counters (fixed emoji set, toggle semantics, partial unique indexes) that fire a screen-wide starburst on increase; emoji throwing is pure ephemeral joy — a whitelisted burst broadcast to the room with a 200 ms server throttle, deliberately never written to the database.

🌐Per-user language

Because Engine 1 stores every curated message bilingually, language is a profile setting, not a channel toggle: the API picks TextEn/TextTr per reader. Turkish and international viewers share one room and each reads it natively.

🔓Open Access flag

Monetization gates (membership + Pro checks across REST and the hub) sit behind a single configuration flag, currently open for the beta. Flipping it re-arms every gate under the same contract — pricing experiments without code changes.

📱PWA install

Manifest + programmatic icon set + iOS standalone meta: Echo installs to the home screen and opens full-screen with a pinned, zoom-locked viewport. No service worker by design — a live-chat product must never serve stale cached rooms.

🧭Operations Copilot (admin)

The admin panel embeds an assistant that researches trending series (live Trakt data), returns real episode tables, deterministic subtitle-source links (built by backend code, never hallucinated), generates per-episode context packs, and answers with exact panel-ready setup recipes — collapsing channel launch to minutes.

12 · Data architecture

One database, honest states

PostgreSQL 18 + pgvector holds everything: identity, catalog (series → episodes → segments), the raw and curated pools with their vectors, channels and their engagement graph (messages, reactions, polls, questions, memberships, visits, preferences). Conventions are uniform — UUIDv7 keys (time-ordered, index-friendly), UTC created/updated stamps, xmin optimistic concurrency, snake_case mapping, foreign keys with deliberate cascade/set-null choices, partial and composite indexes on every hot path. Vector search runs inside the same engine (cosine distance on 1536-d columns), which keeps dedupe and alignment queries transactional with the rows they touch — no external vector store to drift out of sync.

13 · Deployment & operations

Boringly repeatable releases

The whole stack ships as Docker Compose on an Azure VM: caddy (TLS for two public hostnames), web (Next standalone), api, workers, db. A deploy is one idempotent script — git pull, compose build + up, then EF migrations applied against the running database, then a health probe (status/db/time) — executed the same way every time, with logs kept per release. Database evolution is migration-only (30+ applied to date); every schema change ships with its Down path and destructive steps are rehearsed locally first. Failures stay visible by design: pipeline states, per-source cooldowns and structured logs make "why is this comment not showing" a query, not an archaeology dig.

14 · Forward-ready foundations

Already in the schema, waiting for launch day

Two upcoming product directions are pre-wired at the data layer, behind configuration flags that are off today — zero user-visible surface, zero behavioral risk, but launch becomes an endpoint-and-UI exercise instead of a migration project:

🤝Social graph

Friendships (request → accept, single-row pairs, self-request blocked at the DB), external identity mapping and token-based friend invites (register via invite → auto-friendship). Message reply linkage is already indexed with referential integrity — the moderator uses it today; user-to-user replies (public, or private via the existing visible-to-user + personal-group machinery) plug straight in.

🎬Live guest events

Channel events model a scheduled/live/ended guest session — a cast member joining the room on camera — with provider-agnostic stream references (the WebRTC vendor decision is deliberately deferred). Messages can bind to an event, forming the "talk to the guest" lane next to the existing filters, visible to the whole room and the guest.
